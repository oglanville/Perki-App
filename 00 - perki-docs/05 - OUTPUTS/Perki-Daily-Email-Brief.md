# Perki Daily Email — Redesign Brief
### Brand-aligned refresh of the 7am digest · 2026-06-04

A companion to `perki-daily-email.html` (the working, responsive template). The structure you have today is kept; the skin, tone, components and spacing are rebuilt on the new brand. The email is generated by the existing `supabase/functions/daily-digest` edge function, so this is a re-render of that output, not a new pipeline.

## 4. Responsive guidance

- **600px container** centred on the eggshell page, `max-width:600px` so it never stretches on desktop.
- **Fluid down to mobile:** `width:100%` on the container at `max-width:600px`, side padding bumped to 18px via the `.px` class.
- **Stacking:** two-column blocks (Worth adding) and three-column blocks (Where to use) carry a `.stack` class that flips them to full-width single column on phones. Stat tiles drop from four-up to two-up via `.stat{width:50%}`.
- **Touch targets:** every button and row action is at least 34 to 44px tall.
- **Bulletproof basics:** table-based layout, inline styles on every element (media queries are an enhancement, not a dependency), `role="presentation"`, an MSO `PixelsPerInch` fix for Outlook, and a hidden preheader. Avoids background images and anything Outlook drops.
- **Test matrix:** Apple Mail (macOS + iOS), Gmail (web + Android + iOS app), Outlook (Windows + web). Litmus or Email on Acid for a final sweep before go-live.

## 5. Generating the content dynamically (DB mapping)

The function already pulls everything per user. Map each block to its source:

| Block | Source (already queried) |
|-------|--------------------------|
| Dashboard stats | `user_perk_state` (used/dismissed) + derived available; monthly cost from `perks.price` of active tiers; membership count from `user_memberships`. |
| Your memberships | `user_memberships` (provider, membership, tier) joined to the tier price. |
| What to use today | perks where `next_reset_date` is within ~7 days or `reset_period` is weekly, not yet used. |
| Worth adding | providers/tiers in the catalogue the user has not added, ranked by perk count and value. |
| Where to use next | active perks grouped by `category`, top categories by count. |
| All by category | every unlocked perk grouped by `category`, tagged by `feature`. |

Keep the render as small, pure helper functions, one per block, each taking data and returning an HTML string (the function is already shaped this way: `perkRow`, `actionBox`, `buildEmailHtml`). To apply this redesign, swap the old theme constants and the three render helpers:

- Replace the blue divider gradient and `providerColor` fallback (`#0B3D91`) with the brand tokens above.
- Replace `summaryHtml` (green/blue/red tiles) with the four brand stat tiles.
- Replace `perkRow` / `actionBox` / the `<head>` of `buildEmailHtml` with the brand card + button + font styles from the template.
- Add the missing "Worth adding" and "Where to use next" sections as their own helpers if not already split out.

Guard every dynamic list with an empty state (copy in section 2) so the email is never blank.

## 6. Automating the 7am send

The pipeline exists: the `daily-digest` edge function sends via Resend and is triggered by `pg_cron`. To run it cleanly at 07:00 UK time every day:

1. **Schedule in UTC, handle British Summer Time.** `pg_cron` runs in UTC and does not follow DST. 7am London is 06:00 UTC in summer (BST) and 07:00 UTC in winter (GMT). The robust pattern is two cron jobs plus a one-line guard in the function:
   ```sql
   -- runs the function at 06:00 and 07:00 UTC daily
   select cron.schedule('perki-digest-bst', '0 6 * * *', $$ select net.http_post(
     url := 'https://<project>.functions.supabase.co/daily-digest',
     headers := jsonb_build_object('x-cron-secret', '<secret>')) $$);
   select cron.schedule('perki-digest-gmt', '0 7 * * *', $$ select net.http_post(
     url := 'https://<project>.functions.supabase.co/daily-digest',
     headers := jsonb_build_object('x-cron-secret', '<secret>')) $$);
   ```
   In the function, send only when it is actually 7am in London:
   ```ts
   const hour = Number(new Intl.DateTimeFormat('en-GB',
     { timeZone: 'Europe/London', hour: '2-digit', hour12: false }).format(new Date()));
   if (hour !== 7) return new Response('not 7am London', { status: 200 });
   ```
   One of the two crons always lands on 07:00 London; the guard makes the other a no-op. No DST drift, ever.

2. **Keep the existing safeguards:** the `x-cron-secret` header check (already in the handler), and the preview mode for testing to a single address before the real loop.

3. **Idempotency:** record a `digest_sent_at` per user per day (a small table or a column) and skip anyone already sent, so a retry or a double-fire never double-emails.

4. **Deliverability:** confirm SPF, DKIM and DMARC are set for the `perki.app` sending domain in Resend before launch. Keep the plain-text preheader and an unsubscribe link (both present) to stay out of spam.

5. **Send-time data freshness:** the function queries live data at send, so the email always reflects the current `user_perk_state`. No pre-generation needed.

This is purely a scheduling and render-swap change. No new infrastructure.

## 7. Recommended extras

- **Personalised subject** from the value figure ("£186 of perks waiting") lifts open rates and is trivial to compute from the dashboard numbers.
- **One hero action.** If a perk expires today, pull it into a single highlighted strip under the preamble so the most urgent thing is unmissable.
- **Hosted provider logos.** Swap the initials tiles for small hosted PNGs once you have them, for closer parity with the app.
- **Dark-mode safety.** Keep the light palette and add `meta name="color-scheme" content="light"` (already in the template) so clients do not invert it oddly.
